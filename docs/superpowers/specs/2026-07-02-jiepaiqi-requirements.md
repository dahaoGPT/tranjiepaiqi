# jiepaiqi 需求文档

日期：2026-07-02

## 1. 产品定位

`jiepaiqi` 是一个面向独居老人的生活节奏守护系统。它通过安装在水龙头附近的声感传感器识别用水节奏，不采集图像，并保留与采样窗口关联的原始音频片段用于事后复盘，帮助家属或社区志愿者发现并核实可能的异常情况。

系统名称“节拍器”强调两点：

- 学习老人日常生活中稳定、重复的用水节奏。
- 当节奏明显偏离平常时，静默产生异常提醒，提示绑定联系人确认情况。

## 2. 问题背景

独居老人发生跌倒、昏厥、突发疾病等意外时，可能长时间无人发现。摄像头、麦克风等方案虽然能提高监护能力，但会带来较强的隐私侵入感，也容易让老人产生被监视的不适。

用水行为通常与起床、洗漱、做饭、饮水、睡前活动等日常节奏相关。通过水龙头附近的声音特征识别“是否发生用水”，可以在不采集图像的前提下，获得有价值的生活节奏信号。由于系统需要保留原始音频用于复盘，MVP 必须明确音频访问权限、复盘用途和保留期限。第一版音频默认保留 30 天，并且不做语音识别、语音转写或内容理解。

## 3. MVP 目标

第一版 MVP 要实现一个可演示、可测试的端到端闭环：

1. 模拟器和真实设备使用同一接口上报声学特征。
2. 模拟器和真实设备可同时上传与采样窗口关联的原始音频片段。
3. Spring Boot 后端接收、校验、保存声学特征、原始音频文件和原始音频元数据。
4. 后端将连续高置信度特征聚合为用水事件。
5. 后端根据固定规则和个人基线生成异常提醒。
6. Next.js 移动端优先看板展示老人状态、节奏、设备状态和异常记录。
7. 家属或志愿者可确认、解决异常，查看历史记录，并在授权范围内播放关联音频片段进行复盘。

## 4. 用户与角色

第一版采用轻量登录和简单角色绑定。

### 4.1 家属用户

家属用户可查看自己绑定老人的看板、设备状态、用水节奏、异常提醒和历史记录。家属可确认或解决提醒，并记录处理备注。

### 4.2 社区志愿者

志愿者用户可查看分配给自己的老人信息和异常提醒。权限范围与家属类似，但只限于绑定或分配的老人。

### 4.3 老人档案

老人不一定直接使用系统。系统中保存老人姓名、基础备注、联系人、绑定设备、个人节奏基线和异常记录。

### 4.4 设备

设备代表安装在水龙头附近的声感传感器。第一版支持真实设备接入路径，同时提供模拟器作为开发和演示数据源。

## 5. 功能范围

### 5.1 范围内

- Spring Boot 2.7.18 后端应用。
- Next.js 前端应用。
- PostgreSQL 数据库。
- 模块化单体后端结构。
- 声学特征上报接口。
- 原始音频片段上传、保存、关联和复盘播放。
- 原始音频默认保留 30 天，到期后可删除音频文件并保留结构化特征和异常记录。
- 模拟器数据源。
- 真实设备接入接口预留并可按同一协议接入。
- 老人、设备、用户、绑定关系的基础管理能力。
- 移动端优先看板。
- 用水事件聚合。
- 默认规则和个人基线结合的异常判断。
- 异常提醒记录、确认、解决和备注。
- 设备离线状态与老人生活异常区分展示。

### 5.2 范围外

- 摄像头、图像采集、图像识别。
- 长期无限制保存原始音频。
- 语音识别、语音转写、说话人识别或对话内容理解。
- 真实短信、电话、微信、邮件通知通道打通。
- 完整多租户组织权限系统。
- 机器学习异常检测模型。
- 复杂硬件固件开发。
- 医疗诊断或事故结论判断。

## 6. 数据策略

MVP 采用“上传声学特征 + 保留原始音频片段”策略。设备或模拟器上报结构化声学特征，同时上传与采样窗口对应的原始音频片段。后端使用声学特征进行节奏分析和异常判断，原始音频主要用于异常复盘、人工核实和算法调试。

一条声学特征记录包括：

- 设备编号。
- 采样窗口开始时间。
- 采样窗口结束时间。
- 平均音量。
- 峰值音量。
- 低频能量。
- 中频能量。
- 高频能量。
- 水流置信度。
- 关联音频片段编号。

一条原始音频片段记录包括：

- 音频片段编号。
- 设备编号。
- 采样窗口开始时间。
- 采样窗口结束时间。
- 存储路径或对象键。
- 音频格式。
- 时长。
- 文件大小。
- 关联声学特征记录。

后端基于声学特征判断是否存在用水事件。前端默认展示节奏、强度趋势和置信度；在异常详情和授权复盘场景中，可播放与该异常相关的原始音频片段。

## 7. 异常提醒

第一版异常提醒由后端记录，前端展示，不打通真实外部通知通道。外部通知在设计上预留接口，后续可接入短信、邮件、企业微信或微信服务号。

MVP 支持以下异常类型：

- `NO_MORNING_WATER`：老人常见晨间用水窗口结束后仍无用水事件。
- `LONG_CONTINUOUS_FLOW`：连续用水时间过长，可能存在忘关水龙头或异常情况。
- `LOW_DAILY_ACTIVITY`：全天用水次数或总时长显著低于个人基线。
- `DEVICE_OFFLINE`：设备长时间未上报，属于设备状态异常，不直接等同于老人生活异常。

异常提醒应包含原因解释、发生时间、关联老人、关联设备、当前状态、建议动作、处理备注和可复盘的关联音频片段。

## 8. 移动端优先体验

很多家属和志愿者会在手机上查看提醒，因此第一版前端以移动端为主体验。

移动端看板打开后优先显示：

- 当前老人。
- 今日状态。
- 最近一次用水事件。
- 设备在线状态。
- 未处理异常数量。
- 今日节奏时间线。
- 个人基线对比。
- 异常原因和建议动作。
- 异常关联音频复盘入口。

异常详情页要优先回答三个问题：

1. 发生了什么。
2. 为什么系统认为异常。
3. 现在应该做什么。

桌面端只展开同一信息结构，不引入另一套业务流程。

## 9. 验收标准

MVP 完成时应满足：

- 可以启动 PostgreSQL、Spring Boot 后端和 Next.js 前端。
- 模拟器可以持续向后端上报声学特征。
- 后端可以保存特征并聚合出用水事件。
- 正常用水日不会产生异常提醒。
- 晨间无用水场景会生成 `NO_MORNING_WATER`。
- 长时间连续用水场景会生成 `LONG_CONTINUOUS_FLOW`。
- 活动显著偏低场景会生成 `LOW_DAILY_ACTIVITY`。
- 设备停止上报时显示设备离线状态，且不把设备离线误写成老人事故。
- 手机宽度下看板可正常阅读和操作。
- 家属或志愿者可以确认、解决异常并查看历史记录。
- 异常详情页可以播放与异常关联的原始音频片段，用于人工复盘。
- 系统不提供语音识别、语音转写或对话内容分析能力。
- 原始音频默认保留期限为 30 天，超过期限后不影响结构化节奏和异常历史查询。
