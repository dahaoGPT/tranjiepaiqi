# jiepaiqi 设计文档

日期：2026-07-02

## 1. 设计摘要

`jiepaiqi` 第一版采用前后端分离架构。后端是 Spring Boot 2.7.18 模块化单体，数据库使用 PostgreSQL；前端是 Next.js 移动端优先看板。系统通过水龙头附近声感传感器上报的声学特征识别用水事件，并保留采样窗口对应的原始音频片段用于异常复盘。后端根据固定规则和个人基线生成异常提醒。

核心隐私原则是：不采集图像，不做语音识别、语音转写或对话内容理解。MVP 需要保存原始音频片段，但其用途限定为异常复盘、人工核实和算法调试，并通过访问控制和保留期限降低隐私风险。

## 2. 总体架构

```text
Simulator / Sound Sensor
        |
        | acoustic feature batches + raw audio clips
        v
Spring Boot 2.7.18 backend
        |
        | MyBatis / SQL
        v
PostgreSQL
        ^
        |
Next.js mobile-first dashboard
```

后端采用模块化单体，所有模块部署在一个 Spring Boot 应用中。模块之间通过 Java service 接口协作，不通过网络调用拆成微服务。

## 3. 后端模块

### 3.1 identity

负责轻量登录、当前用户、角色和绑定关系。第一版支持家属和志愿者两类使用者。用户只能访问与自己绑定的老人、设备和异常记录。

### 3.2 elder

负责老人档案、联系人、绑定设备和基础备注。老人档案是看板聚合的核心对象。

### 3.3 device

负责设备注册、设备状态、最近上报时间和离线判断。设备异常和老人生活异常必须区分建模。

### 3.4 ingestion

负责接收模拟器或真实设备上报的声学特征和原始音频片段。它校验设备、时间窗口、特征值范围、音频格式、音频时长和文件大小，保存有效数据，并拒绝明显非法请求。

### 3.5 rhythm

负责把连续高置信度声学特征聚合为用水事件，并根据历史事件计算个人基线。

### 3.6 alert

负责异常判断、异常事件持久化、确认、解决和处理备注。第一版生成后端记录，不直接发送真实短信或微信通知。

### 3.7 dashboard

负责为 Next.js 提供聚合后的看板 API，避免前端为一个页面拼接过多底层接口。

### 3.8 audio

负责原始音频片段的元数据管理、存储路径管理、授权播放和复盘访问记录。MVP 可使用后端管理的本地文件存储保存音频文件，在 PostgreSQL 中保存元数据；后续可替换为对象存储。

## 4. 数据模型

核心表包括：

- `users`：登录用户。
- `elders`：老人档案。
- `devices`：设备信息和最近状态。
- `user_elder_bindings`：用户和老人绑定关系。
- `acoustic_features`：声学特征记录。
- `audio_clips`：原始音频片段元数据。
- `water_events`：聚合后的用水事件。
- `rhythm_baselines`：个人节奏基线。
- `alerts`：异常提醒。
- `alert_notes`：异常处理备注。

`acoustic_features` 保存采样窗口级特征数据。`audio_clips` 保存原始音频片段的元数据和存储路径，不直接把大音频内容塞进业务聚合接口。`water_events` 保存业务可读的用水事件。前端主要读取 `water_events`、`rhythm_baselines`、`alerts`、设备状态和授权后的音频复盘链接。

## 5. 数据流

1. 设备或模拟器构造声学特征批次和对应原始音频片段。
2. `ingestion` API 校验设备编号、时间窗口、特征值和音频文件约束。
3. 合法特征保存到 PostgreSQL，音频文件保存到后端管理的音频存储位置，音频元数据保存到 `audio_clips`。
4. `rhythm` 根据连续高置信度特征生成或更新用水事件，并关联相关音频片段。
5. `rhythm` 使用最近 7 天用水事件计算个人基线。
6. `alert` 根据默认规则和个人基线判断异常，并关联异常时间段附近的音频片段。
7. `dashboard` API 聚合老人状态、设备状态、节奏时间线、基线、异常提醒和授权复盘入口。
8. Next.js 移动端看板展示当前状态，用户可确认或解决异常，并可在异常详情页播放关联音频片段。

## 6. 异常判断

第一版采用“固定规则 + 个人基线”。

### 6.1 默认规则

默认规则用于新设备、新老人或历史数据不足时：

- 晨间关键时段结束后无用水，生成 `NO_MORNING_WATER`。
- 连续高置信度用水超过阈值，生成 `LONG_CONTINUOUS_FLOW`。
- 当天用水事件数量或总时长明显偏低，生成 `LOW_DAILY_ACTIVITY`。

### 6.2 个人基线

个人基线使用最近 7 天用水事件计算：

- 常见晨间首次用水窗口。
- 日均用水次数。
- 日均用水总时长。
- 常见活跃时间段。

历史数据不足 7 天时使用默认规则，同时继续积累基线数据。基线只辅助判断，不直接替代可解释规则。

### 6.3 设备离线

设备超过设定时间未上报时产生 `DEVICE_OFFLINE` 状态。该状态在前端单独展示，不作为老人生活异常处理，避免把硬件问题误判为老人事故。

## 7. API 设计

### 7.1 identity API

- `POST /api/auth/login`
- `GET /api/me`
- `GET /api/me/elders`

### 7.2 ingestion API

- `POST /api/devices/{deviceId}/features`
- `POST /api/devices/{deviceId}/audio-clips`

特征请求体包含一个或多个声学特征窗口。音频请求支持上传与采样窗口对应的原始音频片段。两类请求都需要携带设备编号和采样窗口时间，后端用时间窗口和设备编号建立关联。后端返回接收数量、拒绝数量和错误明细。

### 7.3 dashboard API

- `GET /api/dashboard/overview`
- `GET /api/elders/{elderId}/dashboard`
- `GET /api/elders/{elderId}/rhythm`
- `GET /api/elders/{elderId}/baseline`
- `GET /api/elders/{elderId}/device-status`

### 7.4 alert API

- `GET /api/alerts`
- `GET /api/alerts/{alertId}`
- `POST /api/alerts/{alertId}/acknowledge`
- `POST /api/alerts/{alertId}/resolve`
- `POST /api/alerts/{alertId}/notes`
- `GET /api/alerts/{alertId}/audio-clips`
- `GET /api/audio-clips/{audioClipId}/playback`

## 8. 前端设计

Next.js 第一版按移动端优先设计。

页面包括：

- `/login`：轻量登录。
- `/dashboard`：总览，显示绑定老人和当前状态。
- `/elders/[id]`：老人详情，包含节奏趋势、基线、设备和联系人。
- `/alerts`：异常提醒列表。
- `/alerts/[id]`：异常详情，可确认、解决和记录备注。
- `/devices`：设备列表和最近上报状态。

移动端展示一个老人一个主状态，优先显示“是否需要关心”。异常详情页优先呈现异常原因、系统依据、建议动作和音频复盘入口。音频复盘入口默认只出现在异常详情页，不在总览页自动播放。桌面端展开为侧边栏和多列看板，但保持同一业务流程。

## 9. 错误处理

设备上报错误分为拒绝和低置信度两类。

拒绝类错误包括：

- 设备不存在。
- 时间窗口非法。
- 未来时间戳超出允许偏移。
- 必填特征缺失。
- 特征值超出允许范围。
- 音频格式不支持。
- 音频时长超过单个采样窗口允许范围。
- 音频文件大小超过限制。
- 音频片段无法与设备采样窗口关联。

低质量但合法的数据会保存为低置信度，不直接触发异常。这样可以保留调试线索，同时避免噪声造成误报。

前端错误处理要区分：

- 设备离线。
- 数据不足。
- 老人节奏异常。
- 后端暂时不可用。

## 10. 隐私与安全边界

系统默认不采集图像，不做语音识别、语音转写、说话人识别或对话内容理解。MVP 会保存原始音频片段，用于异常复盘、人工核实和算法调试。音频片段必须与设备、采样窗口、老人和异常事件建立明确关联，避免成为无边界的环境录音库。

用户只能访问绑定老人相关数据。音频播放只允许在授权用户查看异常详情或用水事件详情时触发，并应记录访问日志。MVP 默认音频保留期限为 30 天，超过期限后可删除音频文件并保留结构化特征、用水事件和异常记录。异常提醒文案避免制造恐慌，使用“节奏异常，建议确认”等表达，不直接判断事故或健康结论。

## 11. 测试策略

### 11.1 后端单元测试

重点覆盖：

- 声学特征校验。
- 音频片段元数据校验。
- 连续特征聚合为用水事件。
- 7 天数据不足时使用默认规则。
- 晨间无用水生成 `NO_MORNING_WATER`。
- 长时间用水生成 `LONG_CONTINUOUS_FLOW`。
- 低活动生成 `LOW_DAILY_ACTIVITY`。
- 设备离线不等于老人生活异常。

### 11.2 后端 API 测试

覆盖：

- ingestion 接收、拒绝和部分成功。
- 音频上传、关联和授权播放。
- dashboard 聚合数据。
- alert 查询、确认、解决和备注。
- 用户只能访问绑定老人。

### 11.3 前端测试

覆盖移动端关键路径：

- 登录。
- 查看老人总览。
- 查看异常详情。
- 播放异常关联音频片段。
- 确认异常。
- 解决异常。
- 设备离线状态展示。

### 11.4 端到端场景

模拟器提供四类场景：

- 正常日：不生成生活异常。
- 晨间无用水：生成 `NO_MORNING_WATER`。
- 长流水日：生成 `LONG_CONTINUOUS_FLOW`。
- 低活动日：生成 `LOW_DAILY_ACTIVITY`。

端到端测试验证 PostgreSQL、Spring Boot 分析逻辑和 Next.js 展示能串成闭环。

## 12. 实施顺序建议

1. 初始化仓库结构和基础工具链。
2. 建立 PostgreSQL、后端和前端基础配置。
3. 实现后端核心领域模型、音频元数据模型和数据库迁移。
4. 先用测试驱动 `rhythm` 和 `alert` 核心逻辑。
5. 实现 ingestion API、音频上传 API 和模拟器。
6. 实现 dashboard、alert 和音频复盘 API。
7. 实现移动端优先 Next.js 看板与异常音频复盘入口。
8. 串联端到端场景并补齐测试。
